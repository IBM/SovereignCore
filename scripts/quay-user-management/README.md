# Quay Team Setup

This directory contains scripts to automate the creation of teams in Quay organizations, add users to teams, grant read permissions to all repositories, and generate Kubernetes pull secrets.

## Overview

This setup uses a **two-step workflow**:

### Step 1: Create User and Grant Permissions (`create-read-only-account.sh`)
1. Checks if the team user exists, and creates it if not (with auto-generated password)
2. Creates a "sovereign-core-read" team in each Quay organization (or uses existing teams)
3. Adds the specified user to each team
4. Automatically discovers all repositories in each organization
5. Grants read permissions to the team for all repositories in that organization
6. Saves authentication credentials to a JSON file

### Step 2: Generate Pull Secret (`generate-pull-secret.sh`)
1. Uses an application token or OAuth token (created by the user in Quay UI)
2. Generates a Kubernetes pull secret in `dockerconfigjson` format

**Why two steps?**
- The auto-generated password from Step 1 is for initial login only
- Application tokens and OAuth tokens provide better security and can be easily rotated
- Tokens can have specific scopes and expiration dates

## Prerequisites

### Required Tools

- **curl**: HTTP client (usually pre-installed)
- **jq**: JSON processor
  - macOS: `brew install jq`
  - RHEL/Fedora: `sudo dnf install jq`
  - Ubuntu/Debian: `sudo apt-get install jq`
- **base64**: Base64 encoder (usually pre-installed)

### Quay Requirements

- Superuser API token with permissions to:
  - Create users (super:user scope required)
  - Create teams
  - Manage team members
  - List repositories
  - Manage repository permissions
- Access to Quay API endpoint

## Configuration

### Environment Variables

Create a `.env` file from the template:

```bash
cp .env.example .env
# Edit .env with your values
```

#### Required Variables

- **QUAY_URL**: Quay registry URL
  - Example: `https://quay.example.com`
  - Example: `https://registry-quay-quay-enterprise.apps.example.com`

- **TEAM_USER_NAME**: Username to add to teams
  - This user will be added to the "sovereign-core-read" team in each organization
  - If the user does not exist, it will be created automatically with an auto-generated password
  - Example: `readonly-user`

**Note**: User password is automatically generated by Quay API when creating a new user. The generated password will be used for the Kubernetes pull secret.

#### Authentication

- **QUAY_API_TOKEN**: Superuser API token
  - Generate from Quay UI: Account Settings → Application Tokens → Generate Token
  - Requires superuser privileges (super:user scope)

#### Optional Variables

- **CURL_OPTS**: Additional curl options for API calls
  - Useful for SSL certificate issues, timeouts, or proxy settings
  - Example: `--insecure` (skip SSL verification, use only in dev/test)
  - Example: `--connect-timeout 30` (set connection timeout)
  - Example: `--max-time 300` (set maximum operation time)
  - Can combine multiple options: `--insecure --connect-timeout 30`

## Usage

### Complete Workflow (Two Steps)

#### Step 1: Create User and Grant Permissions

```bash
# 1. Prepare environment
cp .env.example .env
vim .env  # Edit with your values

# 2. Load environment variables
source .env

# 3. (Optional) Validate configuration
./create-read-only-account.sh --dry-run

# 4. Run the script
./create-read-only-account.sh
```

**Output**: The script will save authentication credentials to `generated/auth-<team-user-name>.json`

#### Step 2: Create Token and Generate Pull Secret

```bash
# 1. Log in to Quay UI
# Open the URL shown in the script output
# Use credentials from generated/auth-<team-user-name>.json

# 2. Create an application token or OAuth token in Quay UI
# Copy the generated token

# 3. Set one token for pull secret generation
export TEAM_USER_APP_TOKEN=<your-application-token>
# or
export TEAM_USER_OAUTH_TOKEN=<your-oauth-token>

# If both TEAM_USER_APP_TOKEN and TEAM_USER_OAUTH_TOKEN are set,
# generate-pull-secret.sh uses TEAM_USER_OAUTH_TOKEN.

# 4. (Optional) Set custom namespace if needed
# export NAMESPACE=your-namespace  # Default: acm-service-broker

# 5. Generate pull secret
./generate-pull-secret.sh
```

**Output**: The script will generate `generated/pull-secret-<team-user-name>.yaml`

#### Step 3: Apply to Kubernetes

```bash
kubectl apply -f generated/pull-secret-<team-user-name>.yaml
```

```yaml
### Default Permissions Behavior

When the script runs, it configures default permissions for each organization:

**Initial Setup:**
1. Creates or verifies the team exists
2. Adds the user to the team
3. **Sets up default permissions** for the team (new feature)
4. Grants read permissions to all existing repositories

**For New Repositories:**
- When a new repository is created in the organization, Quay automatically grants read access to the `sovereign-core-read` team
- No manual intervention or script re-run is required
- The team user can immediately pull images from new repositories

**For Existing Repositories:**
- The script grants read permissions during execution
- Ensures all repositories (existing and future) are accessible

**Idempotency:**
- If default permission already exists, the script skips creation
- Safe to run multiple times without creating duplicates

  - name: quay-pull-secret-readonly-user
```

## Features

### Automatic Default Permissions

The script automatically configures default permissions for teams in each organization:
- **Sets up default permissions** for newly created repositories
- Automatically grants read access to the team when new repositories are created
- Uses **"Anyone" condition** to apply to all new repositories in the organization
- Ensures both existing and future repositories are accessible to the team
- Skips creation if default permission already exists (idempotent)

**How it works:**
1. After creating/verifying the team, the script checks for existing default permissions
2. If no default permission exists for the team, it creates one with:
   - **Role**: `read`
   - **Delegate**: The team (`sovereign-core-read`)
   - **Condition**: Anyone (applies to all new repositories regardless of creator)
3. When a new repository is created in the organization, Quay automatically grants read access to the team

**Benefits:**
- No manual intervention needed for new repositories
- Consistent access control across all repositories
- Reduces operational overhead for repository management

### Automatic User Creation

The script automatically checks if the team user exists and creates it if necessary:
- Checks user existence before processing organizations
- Creates user with auto-generated password if not found
- Password is automatically generated by Quay API
- Skips user creation if user already exists
- Validates user creation in dry-run mode without actually creating the user
- Provides clear error messages if user creation fails

### Team-Based Access Control

The script creates a "sovereign-core-read" team in each organization:
- Provides centralized access management
- Easier to manage permissions across multiple repositories
- Team members automatically get access to all repositories

### Automatic Repository Discovery

The script automatically discovers all repositories in each organization:
- Supports pagination for large numbers of repositories
- Includes both public and private repositories
- Processes each organization independently

### Existing Team Handling

If a team with the same name already exists:
- The script will use the existing team
- No error is thrown
- User is added to the team if not already a member
- Permissions are still granted to all repositories

### Permission Management

- Grants **read-only** permissions to teams for all repositories
- Skips repositories where team permission already exists
- Continues on errors (with warnings)
- Provides detailed summary of results per organization

### Retry Logic

- Automatic retry for transient failures (up to 3 attempts)
- Exponential backoff between retries
- Handles server errors gracefully

### Error Handling

- Clear error messages for common issues
- Validates environment variables before execution
- Tests API connectivity before making changes
- Continues processing other organizations if one fails
- Provides troubleshooting guidance

## Troubleshooting

### Authentication Failed

**Error**: `Authentication failed. Please check your QUAY_API_TOKEN.`

**Solutions**:
1. Verify your API token is correct
2. Check token hasn't expired
3. Ensure token has superuser privileges (super:user scope)
4. Generate a new token from Quay UI with superuser permissions

### Permission Denied

**Error**: `Permission denied. Please ensure your credentials have admin privileges.`

**Solutions**:
1. Verify your credentials have superuser role
2. Check you have permissions to create users and teams
3. Verify you can manage repository permissions
4. Ensure the user or token has superuser privileges (super:user scope)

### API Connection Failed

**Error**: `Failed to connect to Quay API. Please check QUAY_URL and your credentials.`

**Solutions**:
1. Verify QUAY_URL is correct
2. Check network connectivity to Quay
3. Verify Quay is accessible from your location
4. Check for firewall or proxy issues
5. Verify your authentication credentials are correct

### SSL Certificate Verification Failed

**Error**: `curl: (60) SSL certificate problem: self signed certificate`

**Solutions**:

**Option 1: Use --insecure flag (Development/Test only)**
```bash
export CURL_OPTS="--insecure"
./create-read-only-account.sh
```

**Option 2: Add CA certificate to system trust store (Recommended for Production)**
```bash
# Download the CA certificate
curl -k https://your-quay-url/ca.crt -o quay-ca.crt

# Add to system trust store (varies by OS)
# macOS:
sudo security add-trusted-cert -d -r trustRoot -k /Library/Keychains/System.keychain quay-ca.crt

# RHEL/Fedora:
sudo cp quay-ca.crt /etc/pki/ca-trust/source/anchors/
sudo update-ca-trust

# Ubuntu/Debian:
sudo cp quay-ca.crt /usr/local/share/ca-certificates/
sudo update-ca-certificates
```

**Option 3: Specify CA certificate in curl options**
```bash
export CURL_OPTS="--cacert /path/to/quay-ca.crt"
./create-read-only-account.sh
```

**Security Warning**: The `--insecure` option disables SSL certificate verification and should only be used in development or test environments. For production, always use proper SSL certificates.

### Connection Timeout

**Error**: `curl: (28) Connection timed out`

**Solutions**:
```bash
# Increase connection timeout (default is usually 300 seconds)
export CURL_OPTS="--connect-timeout 60 --max-time 600"
./create-read-only-account.sh
```

### No Organizations Found

**Error**: `No organizations found. This script requires at least one organization.`

**Explanation**: The script requires at least one organization to create teams.

**Solutions**:
1. Verify organizations exist in Quay
2. Check your token has access to view organizations
3. Ensure you're connected to the correct Quay instance

### Team Already Exists

**Info**: `Team already exists`

**Explanation**: This is expected behavior. The script will:
- Use the existing team
- Add the user if not already a member
- Continue to grant permissions
- Not create a duplicate

### Failed to Add User to Team

**Warning**: `Failed to add user to team in <org>`

**Possible Causes**:
- User doesn't exist in Quay
- User credentials are incorrect
- Insufficient permissions to add team members

**Impact**: The script continues processing other organizations

### User Creation Failed

**Error**: `Failed to create team user: <username>`

**Possible Causes**:
- Username is already taken by another user
- Insufficient superuser permissions to create users
- User creation is disabled in Quay configuration

**Solutions**:
1. Verify the username is not already taken
2. Ensure your credentials have superuser permissions (super:user scope)
3. Check if the user already exists
4. Verify Quay allows user creation (some installations may require external authentication)

### Permission Grant Failures

**Warning**: `Failed to grant permission to <org>/<repository>`

**Possible Causes**:
- Repository was deleted during execution
- Insufficient permissions for specific repository
- Network timeout

**Impact**: The script continues processing other repositories

## Security Considerations

### Authentication Credentials Security

- **Never commit** `.env` file to version control
- Store credentials securely (e.g., in a secrets manager)
- Rotate API tokens and passwords regularly
- Use credentials with minimum required permissions
- Prefer API tokens over username/password when possible (more secure and easier to rotate)

### Team User Credentials

- Use a dedicated user account for team membership
- Password is automatically generated by Quay (strong and secure)
- Rotate team user credentials periodically by recreating the user
- Limit the team user's direct permissions (access should come through team membership)
- Store the generated password securely (it's included in the pull secret)

### Team Permissions

- This script grants **read-only** access through teams
- Team members cannot modify or delete repositories
- Consider creating separate teams for different purposes

### Generated Secrets

- Protect generated YAML files (contain credentials)
- Apply secrets only to appropriate namespaces
- Use RBAC to control secret access in Kubernetes
- Rotate team user credentials periodically

## API Endpoints Used

### create-read-only-account.sh

- `GET /api/v1/user/` - Get user information and organizations
- `GET /api/v1/users/{username}` - Check if user exists
- `POST /api/v1/superuser/users/` - Create new user (requires superuser permissions)
- `GET /api/v1/organization/{orgname}/team/{teamname}` - Check team existence
- `PUT /api/v1/organization/{orgname}/team/{teamname}` - Create team
- `PUT /api/v1/organization/{orgname}/team/{teamname}/members/{username}` - Add user to team
- `GET /api/v1/organization/{orgname}/team/{teamname}/members` - List team members
- `GET /api/v1/repository?namespace={orgname}&page={page}` - List organization repositories (with pagination)
- `PUT /api/v1/repository/{namespace}/{repository}/permissions/team/{teamname}` - Grant team permissions
- `GET /api/v1/repository/{namespace}/{repository}/permissions/team/{teamname}` - Check existing team permissions

### generate-pull-secret.sh

This script does not use Quay API endpoints. It only generates a Kubernetes secret using the provided credentials.
